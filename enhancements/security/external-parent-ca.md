# External Parent CA for Platform Certificates

## Summary

This enhancement extends the PKI configuration API introduced by the [Configurable PKI for OpenShift Internal Certificates](/enhancements/security/internal-pki-config.md) enhancement to allow administrators to provide an external CA certificate and key that cross-signs all platform signer certificates. This enables all internally generated certificates to chain to a customer's trusted enterprise root CA while preserving OpenShift's existing internal trust model, automated rotation, and self-healing capabilities.

## Motivation

Enterprise customers in regulated industries (government, finance, telecommunications) are required by security and compliance policies to have all certificates in their infrastructure chain to a trusted enterprise root CA. Currently, OpenShift's internal PKI uses a flat topology where each signer certificate is self-signed. While this model is robust and operationally simple, it means platform certificates cannot be verified against the customer's enterprise certificate hierarchy.

A full external CA integration ŌĆö where an external CA issues every platform certificate ŌĆö was evaluated and rejected as overly complex. It would introduce external dependencies in certificate rotation, compromise OpenShift's automated lifecycle management, and conflict with core reliability design principles.

Instead, this enhancement takes an **intermediate signing CA strategy**: the customer provides a CA certificate and key, and OpenShift cross-signs its existing self-signed signers with it. Internal trust paths remain unchanged, while external verification can trace a chain to the customer's root CA.

### User Stories

* As a security administrator in a government agency, I want all OpenShift platform certificates to chain to my agency's enterprise root CA, so that I can satisfy PKI compliance requirements that mandate a single trust hierarchy across all infrastructure.

* As a platform engineer at a financial institution, I want to provide my organization's CA to OpenShift during installation, so that certificates generated by the platform can be verified against our existing enterprise PKI without requiring manual certificate management.

* As a cluster administrator, I want to add my organization's CA to an existing OpenShift cluster, so that platform certificates are cross-signed on their next rotation cycle and chain to our enterprise root without disrupting running workloads.

* As a compliance auditor, I want to verify that all platform signer certificates have a valid chain to our enterprise root CA, so that I can confirm the cluster meets our organization's PKI policy.

* As an OpenShift SRE, I want the external parent CA cross-signing to be observable through metrics and events, so that I can detect and remediate any failures in the cross-signing process.

### Goals

- Allow administrators to provide an external CA certificate and key that cross-signs all platform signer certificates
- Preserve OpenShift's existing internal trust model ŌĆö self-signed signers remain the primary trust anchors for internal component communication
- Support both install-time (Day-1) and post-install (Day-2) provisioning of the external parent CA
- Cross-sign all signers (regardless of lifetime) on their next rotation cycle for a consistent trust hierarchy
- Leverage the unified library-go certificate creation code path established by the configurable PKI enhancement
- Extend the existing PKI API rather than introducing a new CRD

### Non-Goals

- Replacing OpenShift's internal self-signed signers with externally issued certificates
- Full external CA integration where an external CA issues every certificate
- Validation of the customer-provided CA certificate (trust the customer)
- Automatic renewal or lifecycle management of the customer-provided CA
- PKI resource status conditions for cross-signing state (requires a PKI controller, which does not exist today)
- Alerting on external parent CA certificate expiry
- Per-hosted-cluster external parent CA in HyperShift (management cluster only for this enhancement)
- CSR-based renewal workflows for the external parent CA

## Proposal

This proposal extends the `PKI` resource in `config.openshift.io/v1alpha1` with a new optional `externalParentCA` field within the `PKIProfile`. When configured, the unified library-go certificate creation code generates a cross-signed certificate for every signer alongside the existing self-signed certificate.

At a high level, the changes include:

1. **PKI API Extension**: New `externalParentCA` field in the `PKIProfile` that references a `kubernetes.io/tls` secret in `openshift-config`
2. **Library-go Cross-Signing**: The unified cert creation code generates both self-signed and cross-signed certificates for every signer when an external parent CA is configured
3. **CA Bundle Distribution**: Cross-signed certificates and the external parent CA certificate are appended to existing CA bundle configmaps
4. **Installer Integration**: Support for providing the external parent CA in install-config.yaml, creating the secret and cross-signing Day-1 signers

### Workflow Description

**cluster administrator** is a human user responsible for configuring and managing the OpenShift cluster.

#### Initial Cluster Installation with External Parent CA (Day-1)

1. The cluster administrator obtains a CA certificate and private key from their enterprise PKI team. This may be an intermediate CA, a root CA, or any CA in their hierarchy ŌĆö OpenShift is agnostic to its position in the customer's trust chain.

2. The cluster administrator prepares an install-config.yaml that includes the external parent CA:

```yaml
apiVersion: v1
baseDomain: example.com
metadata:
  name: my-cluster
platform:
  aws:
    region: us-east-1
pki:
  externalParentCA:
    certificate: |
      -----BEGIN CERTIFICATE-----
      ...customer CA certificate...
      -----END CERTIFICATE-----
    privateKey: |
      -----BEGIN RSA PRIVATE KEY-----
      ...customer CA private key...
      -----END RSA PRIVATE KEY-----
  signerCertificates:
    key:
      algorithm: RSA
      rsa:
        keySize: 4096
```

3. The openshift-installer creates the `kubernetes.io/tls` secret in `openshift-config` containing the external parent CA certificate and key.

4. The installer generates all signer certificates using the unified library-go code path. Because the external parent CA is available, library-go generates both the self-signed certificate and a cross-signed certificate for each signer.

5. The installer creates the initial `PKI` custom resource with the `externalParentCA` field referencing the secret name.

6. Cross-signed certificates and the external parent CA certificate are included in CA bundles from the start.

#### Adding an External Parent CA Post-Installation (Day-2)

1. The cluster administrator creates a `kubernetes.io/tls` secret in `openshift-config`:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: customer-ca
  namespace: openshift-config
type: kubernetes.io/tls
data:
  tls.crt: <base64-encoded CA certificate>
  tls.key: <base64-encoded CA private key>
```

2. The cluster administrator edits the PKI resource to reference the secret:

```bash
oc edit pki cluster
```

```yaml
apiVersion: config.openshift.io/v1alpha1
kind: PKI
metadata:
  name: cluster
spec:
  certificateManagement:
    mode: Custom
    custom:
      defaults:
        key:
          algorithm: RSA
          rsa:
            keySize: 2048
      externalParentCA:
        name: customer-ca
```

3. Operators detect the PKI resource change on their next sync.

4. On each signer's next rotation cycle, library-go generates the cross-signed certificate alongside the self-signed certificate and appends both the cross-signed cert and the external parent CA cert to CA bundles.

5. Until rotation happens, signers remain self-signed only ŌĆö no immediate disruption to the running cluster.

#### Updating the External Parent CA

1. The cluster administrator updates the secret in `openshift-config` with a renewed CA certificate and key.

2. Operators detect the secret change on their next sync.

3. On each signer's next rotation cycle, library-go generates a new cross-signed certificate using the updated external parent CA.

#### Removing the External Parent CA

1. The cluster administrator removes the `externalParentCA` field from the PKI resource.

2. On each signer's next rotation cycle, library-go stops generating cross-signed certificates.

3. Cross-signed certificates and the external parent CA certificate are removed from CA bundles during bundle reconciliation.

### API Extensions

This enhancement extends the existing `PKI` Custom Resource Definition in `config.openshift.io/v1alpha1`. No new CRDs are introduced.

#### PKI Resource Extension

The `PKIProfile` type (used within the `Custom` certificate management mode) gains a new optional field:

```go
type PKIProfile struct {
    // defaults specifies the default cryptographic parameters for all certificates.
    // +required
    Defaults DefaultCertificateConfig `json:"defaults"`

    // externalParentCA references a kubernetes.io/tls secret in the openshift-config
    // namespace containing a CA certificate and private key. When set, all platform
    // signer certificates are cross-signed by this CA, enabling certificate chain
    // verification against the customer's enterprise root CA.
    // +optional
    ExternalParentCA *ExternalParentCAReference `json:"externalParentCA,omitempty"`

    // signerCertificates overrides cryptographic parameters for signer certificates.
    // +optional
    SignerCertificates *CertificateConfig `json:"signerCertificates,omitempty"`

    // servingCertificates overrides cryptographic parameters for serving certificates.
    // +optional
    ServingCertificates *CertificateConfig `json:"servingCertificates,omitempty"`

    // clientCertificates overrides cryptographic parameters for client certificates.
    // +optional
    ClientCertificates *CertificateConfig `json:"clientCertificates,omitempty"`
}

type ExternalParentCAReference struct {
    // name is the name of a kubernetes.io/tls secret in the openshift-config namespace.
    // The secret must contain tls.crt (CA certificate) and tls.key (CA private key).
    // The certificate may include the full chain up to the root CA.
    // +required
    Name string `json:"name"`
}
```

#### Compatibility Level

This API extension follows the same compatibility level as the parent PKI API:
- **v1alpha1 (Level 4, TechPreviewNoUpgrade):** During development
- **v1 (Level 1, GA):** Target OpenShift release

The `externalParentCA` field is gated behind the same `ConfigurablePKI` feature gate.

### Topology Considerations

#### Hypershift / Hosted Control Planes

For this enhancement, the external parent CA is supported on the **management cluster only**. The PKI resource on the management cluster can reference an external parent CA, and all management cluster signers (including those managed by the `control-plane-pki-operator`) are cross-signed.

Per-hosted-cluster external parent CA support is deferred to a future enhancement.

#### Standalone Clusters

This enhancement is fully applicable to standalone clusters. All platform signer certificates are cross-signed by the external parent CA.

#### Single-node Deployments or MicroShift

**Single-Node OpenShift (SNO):**
- Fully supported with the same API and behavior as multi-node clusters
- Cross-signing adds minimal overhead ŌĆö one additional certificate per signer, generated only during rotation

**MicroShift:**
- The external parent CA should be exposed through MicroShift's YAML configuration file:

```yaml
apiVersion: v1alpha1
kind: MicroShiftConfig
pki:
  certificateManagement:
    mode: Custom
    custom:
      externalParentCA:
        certificate: |
          -----BEGIN CERTIFICATE-----
          ...
          -----END CERTIFICATE-----
        privateKey: |
          -----BEGIN RSA PRIVATE KEY-----
          ...
          -----END RSA PRIVATE KEY-----
```

### Implementation Details/Notes/Constraints

#### Cross-Signing Mechanics

When `externalParentCA` is configured, the unified library-go certificate creation code performs the following for every signer certificate:

1. **Normal signer creation (unchanged):** Generate key pair using PKI profile parameters. Create self-signed certificate. Store in the existing secret.

2. **Cross-signed certificate (new):** Using the same public key from step 1, create a second certificate signed by the external parent CA. The cross-signed certificate has the same subject, SANs, and extensions as the self-signed certificate ŌĆö only the issuer and signature differ. The cross-signed certificate's validity period is bounded by the shorter of: the signer's own lifetime, or the external parent CA's remaining validity.

3. **CA bundle distribution (modified):** Append the cross-signed certificate and the external parent CA certificate to existing CA bundle configmaps. Verifiers can now build two trust paths:
   - **Internal path:** leaf cert ŌåÆ self-signed signer (existing, unchanged)
   - **External path:** leaf cert ŌåÆ cross-signed signer ŌåÆ external parent CA ŌåÆ customer's root

Because the self-signed certificate and cross-signed certificate share the same key pair, all existing leaf certificates (serving, client) remain valid under both trust paths without any changes.

On rotation, when a signer rotates to a new key pair, library-go generates both the new self-signed certificate and the new cross-signed certificate in one operation. Any previous cross-signed certificate for that signer is replaced ŌĆö only the current cross-signed certificate is retained in CA bundles.

#### All Signers Are Cross-Signed

All platform signers are cross-signed regardless of lifetime (10-year, 1-year, 1-day). This provides a consistent trust hierarchy for compliance and audit purposes. Short-lived signers rotate frequently, but the cross-signing operation is lightweight (one additional signature per rotation).

#### No Validation of the External Parent CA

OpenShift does not validate the customer-provided CA certificate. If the certificate is invalid (expired, missing CA:TRUE basic constraint, incompatible algorithm), cross-signing will fail and operators will report errors through normal event and logging mechanisms. This is consistent with the design philosophy of trusting the customer's PKI decisions.

#### Secret Storage

The external parent CA certificate and private key are stored as a standard `kubernetes.io/tls` secret in the `openshift-config` namespace. This follows the same pattern used for user-provided serving certificates (`APIServer.spec.servingCerts`).

The `tls.crt` field may contain the full certificate chain (intermediate + root) to facilitate chain verification by external parties.

### Risks and Mitigations

**Risk: Compromised external parent CA private key**

*Mitigation:* The private key is stored in a Kubernetes secret with standard RBAC protections. The same risk exists for all secrets in `openshift-config`. Customers with strict key management requirements should use short-lived intermediate CAs and rotate them regularly. The customer is responsible for the security of their CA material.

**Risk: External parent CA certificate expires, breaking chain verification**

*Mitigation:* Cross-signing failure does not affect internal cluster operation ŌĆö self-signed signers remain the primary trust anchors. Only external chain verification is affected. The customer is responsible for updating the secret with a renewed CA before expiry. Future work may add alerting on expiry.

**Risk: Cross-signed certificates increase CA bundle size**

*Mitigation:* Each signer adds one additional certificate (the cross-signed version) to its CA bundle. With approximately 50 signers, this adds roughly 50 certificates across all bundles. Certificate data is small (typically 1-2 KB each), so the total increase is negligible.

**Risk: Performance impact of cross-signing during rotation**

*Mitigation:* Cross-signing is a single additional signature operation per signer rotation. Even with RSA 4096, signing takes milliseconds. This is negligible compared to the key generation that already occurs during rotation.

### Drawbacks

**Increased Complexity:** This feature adds cross-signing logic to the library-go certificate creation code. However, the logic is straightforward (sign the same public key with a different CA) and is centralized in one code path.

**Private Key in Cluster:** The external parent CA's private key must be stored in the cluster as a Kubernetes secret. Some organizations may be uncomfortable with this. However, this is the same trust model used for all Kubernetes secrets, and the alternative (external signing via CSR workflows) would introduce the complexity and external dependencies that this enhancement explicitly avoids.

**Customer Responsibility for CA Lifecycle:** OpenShift does not manage the external parent CA's lifecycle (renewal, revocation). This is by design ŌĆö the customer owns their enterprise PKI ŌĆö but it means the customer must maintain the CA independently.

## Alternatives (Not Implemented)

### Alternative 1: Full External CA Integration

Have an external CA issue every platform certificate directly, replacing OpenShift's internal PKI entirely.

**Not selected because:**
- Introduces external dependencies in certificate rotation and bootstrapping
- Compromises OpenShift's automated lifecycle management and self-healing
- Requires network connectivity to an external CA at all times
- Conflicts with core reliability design principles
- Explicitly rejected in the scope rationale of OCPSTRAT-2029

### Alternative 2: Separate CRD (ExternalCA)

Create a new CRD in `config.openshift.io` specifically for external CA configuration.

**Not selected because:**
- Fragments the PKI configuration across two cluster-scoped resources
- Requires operators to watch an additional resource
- The external parent CA is fundamentally a PKI configuration concern and belongs in the PKI resource

### Alternative 3: Convention-Based Secret Detection

No API changes. If a well-known secret (e.g., `openshift-config/external-parent-ca`) exists, library-go automatically cross-signs.

**Not selected because:**
- No declarative API for auditing or automation
- "Magic secret" pattern is fragile and undiscoverable
- Cannot express intent or report status through the API
- No way to distinguish "not configured" from "misconfigured"

## Open Questions

1. Should the `externalParentCA` field be available in `Default` mode (not just `Custom` mode)? An administrator may want cross-signing without overriding any cryptographic parameters. This would require adjusting the discriminated union or allowing `externalParentCA` at the `certificateManagement` level rather than inside the `custom` profile.

2. What is the interaction between cross-signed certificates and the existing cross-signing that occurs during signer rotation? When a signer rotates, the old signer is cross-signed by the new signer for trust continuity. The external parent CA cross-signing is a separate concern (chain to enterprise root) but both produce additional certificates for the same signer key.

## Upgrade / Downgrade Strategy

**Upgrade:** Clusters upgrading to a version with this feature gain the `externalParentCA` field in the PKI API. If the field is not configured, behavior is identical to before the upgrade. No action is required to maintain previous behavior.

**Downgrade:** If a cluster is downgraded from a version with `externalParentCA` support to one without:
- The `externalParentCA` field in the PKI resource becomes unrecognized and is preserved but ignored (standard Kubernetes behavior for unknown fields with structural schemas)
- Cross-signed certificates remain in CA bundles but are not refreshed on rotation ŌĆö they will naturally expire
- Self-signed certificates continue to function as primary trust anchors ŌĆö no disruption to internal cluster operation
- The secret in `openshift-config` is not deleted

## Version Skew Strategy

During an upgrade, some operators may have the cross-signing capability while others do not yet:
- Operators without cross-signing support continue to generate only self-signed signer certificates ŌĆö this is the existing behavior and is safe
- Operators with cross-signing support generate both self-signed and cross-signed certificates
- The cluster operates correctly in this mixed state because internal trust paths (self-signed) are unchanged
- Cross-signing converges as operators are upgraded ŌĆö no coordination required

## Operational Aspects of API Extensions

This enhancement extends the existing PKI CRD with one new optional field. The operational impact is minimal:

- **No new webhooks or aggregated API servers**
- **No new finalizers**
- **CRD change:** One additional optional field (`externalParentCA`) in the `PKIProfile` type
- **Secret dependency:** Operators must be able to read the referenced secret in `openshift-config`. This uses the same access pattern as existing user-provided certificate secrets.
- **Failure mode:** If the referenced secret does not exist or is invalid, cross-signing fails silently for that rotation cycle. Self-signed certificates are unaffected. Operators log errors and emit events.
- **No impact on existing SLIs:** Cross-signing adds one signature operation per signer rotation, which is negligible.

## Support Procedures

**Symptom: Cross-signed certificates are not appearing in CA bundles**

1. Verify the secret exists: `oc get secret -n openshift-config <secret-name>`
2. Verify the PKI resource references the secret: `oc get pki cluster -o yaml`
3. Check operator logs for cross-signing errors: `oc logs -n openshift-kube-apiserver-operator deploy/kube-apiserver-operator | grep -i cross-sign`
4. Cross-signing occurs on signer rotation ŌĆö verify the signer has rotated since the external parent CA was configured

**Symptom: Certificate chain verification fails against enterprise root**

1. Verify the external parent CA certificate includes the full chain: `openssl x509 -in <cert> -text -noout`
2. Verify the cross-signed certificate's issuer matches the external parent CA subject
3. Verify the external parent CA has not expired

**Disabling the feature:**

Remove the `externalParentCA` field from the PKI resource. On next signer rotation, cross-signed certificates are no longer generated. Existing cross-signed certificates in CA bundles are removed during reconciliation. Internal cluster operation is not affected at any point.

No new infrastructure is needed. This enhancement uses existing CI infrastructure and test frameworks.
